Case Study 02 · Design Systems

Building a design system from scratch.

Establishing a shared design language across product, marketing, and client-facing surfaces

Product
EffiGO eProcurement
My Role
Lead Designer
Scope
Foundations + Component Library

01 — Context

No system, no shared language

EffiGO is an enterprise eProcurement platform serving clients across sourcing, procurement, dispatch, and contract management. As the product grew across modules, so did the inconsistency — each area developed its own visual patterns, component styles, and interaction logic independently.

The problem wasn't isolated to the product. Marketing materials, client presentations, and the platform itself all looked like they came from different companies. There was no brand system, no shared component set, and no common language between designers and developers.

I helped design the system from the ground up — establishing foundations, designing the component library, and working directly with the development team through handoff.

02 — The Problem

One product, no shared visual logic

The inconsistency manifested differently across surfaces but came from the same root cause: every team was making design decisions independently, with no reference point. The cost showed up in multiple ways.

Each module used its own layout logic, spacing rules, and component styles — making the product feel like a collection of separate tools rather than a unified platform. Dashboards, tables, and configuration-heavy forms followed different visual and interaction patterns, so users had to relearn the interface each time they moved between modules.

Complex admin and workflow screens became dense and hard to scan because no hierarchy principles were applied consistently. Information sat at equal visual weight regardless of priority.

Beyond the product, marketing decks and client presentations looked visually disconnected from what clients actually used day to day. There was no brand system tying surfaces together.

The development cost
Without a shared component library, the same UI patterns were being rebuilt from scratch across modules — time spent on solved problems rather than new ones.
03 — Approach

Principles before pixels

Before building a single component, I established the principles that would govern every decision. For a data-heavy enterprise product, the constraints were specific: components had to work across dense admin tables and sparse empty states, across tight and wide viewports, and for both technical and non-technical users.

The sequencing was deliberate: foundations first — colour, typography, spacing — then components. Without a stable foundation, every component decision becomes a negotiation. With it, components almost design themselves.

Principle 01
Clarity over decoration
Every visual decision had to earn its place by aiding comprehension. Nothing decorative that didn't carry meaning.
Principle 02
Density without noise
Enterprise workflows require dense UIs. The system had to support high information density without creating visual chaos — structure through hierarchy, not decoration.
Principle 03
Predictable patterns
Users moving between modules should never have to relearn how something works. Consistency reduces cognitive load across the whole product.
Principle 04
Built for handoff
Every component needed to be documented well enough that a developer could implement it without a design walkthrough. The system had to work without me in the room.
04 — Foundations

A shared language for the whole product

Foundations are the decisions that everything else inherits from. Getting these right meant every component built on top of them would automatically feel consistent — without anyone having to consciously enforce it.

Colour

A primary navy anchors the brand identity across product and marketing. A semantic status layer — green for success, red for error, amber for warning, a distinct burgundy for rejection — gives meaning to state across every module. These were not decorative choices; in a procurement workflow, the difference between a warning and a rejection carries real business consequence, and the colour system had to reflect that.

A calibrated neutral text scale supports readability at every density level — from large dashboard headings down to 10px table metadata. Background and surface colours were chosen specifically for dense admin interfaces where users spend hours at a time.

Typography — Manrope + Poppins

Two typefaces, each with a clear role. Manrope Extrabold handles headings and module titles — bringing structural weight to dashboards and navigation. Poppins covers all body text, form labels, table data, and metadata, with a weight range from Regular through Bold.

The type scale runs from 32px headings to 10px metadata, with every size and weight combination named and documented — so a developer could implement from spec without approximating from visual inspection.

Spacing and layout

A standardised spacing system replaced the ad hoc padding and margin decisions that had made screens inconsistent. Consistent component spacing, section padding, and grid behaviour created predictable layouts across modules — reducing the cognitive effort of moving between different parts of the product.

05 — Components

Every state, every edge case

The hardest part of building components for an enterprise system isn't the default state — it's everything else. Empty, loading, error, disabled, focused, filled, indeterminate. Every component had to handle the full range of states that a complex procurement workflow would surface.

Full component documentation cannot be shared due to NDA. Below is an overview of scope and coverage.

Text fields
Single-line and multi-line variants. Default, focused, filled, disabled, and error states — each with and without helper text. Character counters for length-constrained inputs.
Dropdowns
Single select, multi-select with inline checkboxes, grouped options, and action menu variants. Each handling default, open, hover, selected, and disabled states. Behaviour aligned with how developers would implement it, not just how it looks in isolation.
Banners
Four semantic variants — default, warning, success, error — using status colours from the foundation. Dismissible, with consistent icon treatment. Used across the product for system messages, workflow alerts, and validation feedback.
Notifications
With and without user avatar, compact and expanded layouts, with dismiss and action link variants. Designed to work in both panel and inline contexts across different modules.
Tags + status indicators
Progress indicators with percentage rings, status message tags (approved, rejected, toast notifications), and dismissible action tags with icon support. Used extensively across approval workflows and supplier management screens.
Navigation
Sidebar in collapsed and expanded states. Active, hover, and default item variants. Icon-only mode for high-density layouts. Expandable sub-navigation for modules with deep hierarchies. Designed to accommodate the platform's 15+ module structure without visual overwhelm.
File uploads
Empty drop zone, uploading state with progress bar, uploaded state with preview and delete, error state with retry — with and without template download option. Procurement workflows involve heavy document handling, so this component needed to cover every real state users encounter.
Radio, checkbox + toggles
Full state coverage including indeterminate states for bulk selection in tables. Toggle on and off states. Notification badge with unread count. The most used interaction patterns in configuration-heavy admin screens.
Date picker
Calendar view with selected date, today highlight, and range selection states. Date chip variants for selected, active, and disabled. Designed for submission deadline and contract date inputs across the sourcing flow.
Tooltips
Plain text, with header, and data summary variants — the latter designed specifically for pricing breakdown tooltips in procurement tables. All four directional positions documented.
Workflow builder
The most complex category. Connector lines with directional variants, condition builder patterns, flow type selectors, and approval chain configurators. These underpin the platform's conditional workflow engine — designed so non-technical users can configure complex approval logic without engineering support.
06 — Developer Collaboration

Built with developers, not for them

Component feasibility was aligned with the development team before and during system creation — not after. This meant every component was technically viable before it was finalised in Figma, and developers understood the intent behind each state and variant rather than interpreting it post-handoff.

Each component was reviewed collaboratively during handoff, with developer feedback incorporated into the final specs. Files were organised for clear implementation — consistent naming conventions, documented states, and grouped variants that matched how developers would actually structure the code.

The team's response was immediate. Having a fixed reference meant they could concentrate on complex business logic rather than making visual decisions that should have been resolved in design. Ambiguity in handoff dropped significantly. Implementation became faster and more consistent.

07 — Impact

From scattered patterns to a shared framework

What began as inconsistency across every surface evolved into a framework that simplified decisions and reduced friction — for designers, developers, and clients alike.

  • Product consistency — all new modules built using system components, no new one-off patterns introduced after launch
  • Faster development — developers working from a shared library rather than rebuilding from scratch per module
  • Brand coherence — marketing and client presentations unified with the product for the first time
  • Reduced design rework — decisions made at system level rather than per screen, freeing designers to focus on harder problems
  • Scalable foundation — new components and modules inherit foundations automatically, so the system grows without breaking existing patterns
  • Developer confidence — reduced ambiguity in handoff, implementation decisions no longer left to interpretation

The design system was not a deliverable — it was infrastructure. The real measure of success was that it became invisible: designers stopped debating component decisions, developers stopped improvising visual logic, and the product started feeling like a single coherent tool rather than a collection of modules.

If given more time: I would have introduced a formal token layer — separating primitive values from semantic assignments to make theming and future brand updates easier to propagate across the system without touching individual components.
